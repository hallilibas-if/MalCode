# Minimal Double-Entry Ledger

This repository contains a clean, from-scratch implementation for Mal's Head of Engineering take-home assessment. The current milestone is the PostgreSQL data model and schema for a minimal double-entry ledger.

The folders `blnk-main/` and `double-entry-bank-Go-main/` are local reference projects only. They are not part of this implementation and should not be copied from.

## Current Status

Milestone 1 is complete:

- `schema.sql`: PostgreSQL schema for accounts, immutable ledger entries, and idempotency keys.

The Go application will be added in later milestones. For now, the runnable artifact is the schema.

## Requirements

- PostgreSQL 14 or later
- `psql`
- Optional: Docker, if you do not already have PostgreSQL running locally

## Install PostgreSQL On Ubuntu

Use this path when you want PostgreSQL installed directly on Ubuntu instead of running it through Docker.

Update package metadata:

```bash
sudo apt update
```

Install PostgreSQL and common extra utilities:

```bash
sudo apt install -y postgresql postgresql-contrib
```

Start PostgreSQL and enable it on boot:

```bash
sudo systemctl start postgresql
sudo systemctl enable postgresql
```

Check that the service is running:

```bash
sudo systemctl status postgresql
```

Confirm the installed client version:

```bash
psql --version
```

Create a local database user for this project. The command below prompts for a password:

```bash
sudo -u postgres createuser --interactive --pwprompt mal_ledger_user
```

When prompted:

- Answer `no` to superuser, database creation, and role creation for a normal local setup.
- The database will be created by the built-in `postgres` admin user and owned by this project user.

Create the database owned by that user:

```bash
sudo -u postgres createdb --owner=mal_ledger_user mal_ledger
```

Set the local connection string:

```bash
export DATABASE_URL="postgres://mal_ledger_user:YOUR_PASSWORD@localhost:5432/mal_ledger?sslmode=disable"
```

Apply the schema:

```bash
psql "$DATABASE_URL" -f schema.sql
```

If `CREATE EXTENSION pgcrypto` fails because your local PostgreSQL policy restricts extensions, apply the schema once as the `postgres` admin user or ask a database admin to enable `pgcrypto` in the `mal_ledger` database.

Verify the tables:

```bash
psql "$DATABASE_URL" -c "\dt"
```

If local password authentication fails, check `/etc/postgresql/*/main/pg_hba.conf` and make sure local TCP connections use password authentication such as `scram-sha-256` or `md5`, then restart PostgreSQL:

```bash
sudo systemctl restart postgresql
```

## Smoke Test The Schema

The following script creates two accounts, posts one double-entry transaction, calculates balances from ledger entries, and records an idempotency response.

```bash
psql "$DATABASE_URL" <<'SQL'
WITH
source_account AS (
    INSERT INTO accounts (owner_id, currency)
    VALUES ('customer_a', 'USD')
    RETURNING id
),
destination_account AS (
    INSERT INTO accounts (owner_id, currency)
    VALUES ('customer_b', 'USD')
    RETURNING id
),
new_transaction AS (
    SELECT gen_random_uuid() AS id
)
INSERT INTO entries (account_id, transaction_id, amount, type)
SELECT source_account.id, new_transaction.id, 2500, 'DEBIT'
FROM source_account, new_transaction
UNION ALL
SELECT destination_account.id, new_transaction.id, 2500, 'CREDIT'
FROM destination_account, new_transaction;

SELECT
    account_id,
    SUM(CASE WHEN type = 'CREDIT' THEN amount ELSE -amount END) AS balance_minor_units
FROM entries
GROUP BY account_id
ORDER BY account_id;

INSERT INTO idempotency_keys (
    key,
    request_hash,
    response_status_code,
    response_payload,
    completed_at
)
VALUES (
    'demo-key-001',
    'sha256:demo-request',
    201,
    '{"transaction_posted": true}'::jsonb,
    now()
);
SQL
```

Expected result:

- One account has `-2500`.
- One account has `2500`.
- Reusing `demo-key-001` should fail because the idempotency key is the primary key.

## Schema Design

### `accounts`

Represents a single-currency financial account.

Key fields:

- `id`: UUID primary key generated by PostgreSQL.
- `owner_id`: external owner identifier. This is intentionally `TEXT` so the ledger is not coupled to a specific users table.
- `currency`: three-letter uppercase ISO-4217 code.
- `created_at`: immutable creation timestamp.

### `entries`

The immutable double-entry core. Each logical transaction is represented by two rows with the same `transaction_id`:

- one `DEBIT`
- one `CREDIT`

Key constraints:

- `amount BIGINT`: stores minor units, such as cents, so financial math avoids float rounding errors.
- `amount > 0`: negative amounts are rejected because direction must be modeled by `type`.
- `type IN ('DEBIT', 'CREDIT')`: only valid accounting sides are allowed.
- `UNIQUE (transaction_id, type)`: prevents accidentally inserting two debits or two credits for the same minimal transaction.

Balances are derived from entries:

```sql
SELECT
    account_id,
    SUM(CASE WHEN type = 'CREDIT' THEN amount ELSE -amount END) AS balance_minor_units
FROM entries
WHERE account_id = $1
GROUP BY account_id;
```

### `idempotency_keys`

Stores client retry keys and cached responses.

Key fields:

- `key`: client-supplied primary key.
- `request_hash`: canonical hash of the request payload, used to reject key reuse with a different request.
- `response_status_code`: cached HTTP status after completion.
- `response_payload`: cached JSON response after completion.
- `completed_at`: marks the request as replayable.

## Future Go Implementation Contract

Use these names and responsibilities in later milestones so the code remains easy to review.

### Domain Types

`Account`

- Represents one ledger account.
- Fields should map directly to the `accounts` table.

`Entry`

- Represents one immutable ledger row.
- Must always be single-sided: either debit or credit.

`Transaction`

- Represents one logical movement between accounts.
- In the minimal implementation, one transaction creates exactly two entries.

`IdempotencyRecord`

- Represents one row in `idempotency_keys`.
- Used to cache and replay completed responses.

### Repository Functions

`CreateAccount(ctx, ownerID, currency)`

- Inserts one account.
- Validates currency before sending data to PostgreSQL.

`PostTransaction(ctx, fromAccountID, toAccountID, amount, idempotencyKey)`

- Starts one database transaction.
- Reserves the idempotency key.
- Locks or validates both accounts.
- Inserts one debit and one credit entry with the same `transaction_id`.
- Stores the cached idempotency response.
- Commits once all writes succeed.

`FetchBalance(ctx, accountID)`

- Calculates balance dynamically from `entries`.
- Does not trust a mutable cached balance column.

`GetEntriesByTransaction(ctx, transactionID)`

- Returns the two entry rows linked by `transaction_id`.
- Useful for debugging, audit, and future reconciliation.
